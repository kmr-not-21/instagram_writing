# 花屋Moja Instagram文面作成プロジェクト

このリポジトリは、神奈川県藤沢市の花屋「Moja」のInstagram投稿（[@hana.moja_](https://www.instagram.com/hana.moja_/)）の文面を、Claude Codeと依頼者が共同編集して作成するためのワークスペースです。

店のコンセプト：**「花とグリーン、どちらも主役級。特別なかけあわせ。」**

---

## 依頼者プロファイル

- 花屋Moja（神奈川県藤沢市）の店主
- 自家染めガーベラなど、独自の技術・こだわりを持つ
- 花材や配置に関する専門知識が豊富（品種名、束ね方の技法など）
- AI生成の文面をそのまま採用せず、要素を取捨選択して自分の言葉で再構成するスタイル
- TODOコメント・FBコメントを文面中に埋めて方向性を示す編集スタイルを取る

## ディレクトリ構造

```
/
├── CLAUDE.md                    # このファイル（プロジェクト全体ガイド）
├── rules/                       # 執筆ルール・リクエストテンプレート
│   ├── writing_rules.md         # 通常投稿の文体・段落構成・トーン
│   ├── calendar_writing_rules.md # カレンダー投稿の文体・段落構成・トーン
│   ├── request.md               # 通常投稿のリクエストテンプレート
│   ├── calendar_request.md      # カレンダー投稿のリクエストテンプレート
│   └── self_check.md            # 初稿提出前のセルフチェックリスト
├── draft/{{投稿日}}/              # 検討中の文面・写真・request.md
│   ├── request.md               # 依頼者が記入する投稿情報＋Question/Answer履歴
│   ├── content.txt              # AIが書き出す検討過程と複数パターン（全パターン保持）
│   └── *.jpg                    # 添付写真
├── posted/                      # 確定投稿
│   ├── normal_post/{{日付}}.txt  # 通常投稿の確定版
│   └── calendar_post/*.txt      # カレンダー投稿の確定版
└── prompt/, inception/          # 過去の検討資料
```

## エントリポイント

依頼者から「`draft/{{投稿日}}/request.md`を実行してください」と指示された場合、以下の手順で対応する：

1. `draft/{{投稿日}}/request.md` を読む
2. ファイル先頭の参照先（`rules/writing_rules.md` または `rules/calendar_writing_rules.md`）を確認
3. 通常投稿なら `rules/request.md` のワークフロー、カレンダー投稿なら `rules/calendar_request.md` のワークフローに従う
4. **初稿執筆前に必ず `rules/self_check.md` の項目を確認する**

---

## 執筆ワークフローの大原則

### 1. 重要な決定は独自に行わず、必ず[Question]で確認する

写真の構成、テーマのキーワード、用途への言及度合い、品種の同定など、依頼者の知識・判断が必要な事項は、初稿執筆前に `draft/{{投稿日}}/request.md` 下部に **[Question]タグ＋推奨案＋空の[Answer]** で提示する。

**Question/Answerフォーマット**：1つの質問には1つの[Answer]欄を用意（[Question 1]→[Answer 1] の番号付きペア）。複数の質問を1つの[Answer]にまとめない（依頼者が個別回答できるようにするため）。

### 2. AI初稿は「素材」、依頼者が再構成する前提で書く

依頼者は2パターン提案をそのまま採用せず、要素を取捨選択して**独自のC案**を作る。AIパターンの役割は「最終稿」ではなく「素材・叩き台」。構成の骨格・キーワード・描写の切り口を提供する。

**依頼者の典型的な加筆傾向**：
- 括弧書きの補足で店主らしい人柄・小ネタを足す
- 五感に訴える具体的な体験提案を足す
- AIの整いすぎた表現を崩して自然な語り口にする

**典型的な減筆傾向**：詳細は `rules/self_check.md` を参照。

### 3. TODO付き返却には文脈に合った表現で埋める

依頼者が自分で書いた文面に `{TODO: ...}` や `(FB: ...)` を残して校正・レビューを依頼するケースがある。TODOの意図を読み取り、複数案を提示してから埋める。FBは「方向転換の指示」として捉え、現状の問題点を整理してから書き直す。

### 4. 初稿の完成度を引き上げ、TODO往復サイクルを最小化する

AI初稿 → 依頼者がTODOを埋めて返却 → AI流し込み、という差し戻しは往復コストが高い。これを避けるため：
- 写真確認・テーマ表現・切り口判断などの**判断系は事前ステップの[Question]/[Answer]で完結**させる
- **過去投稿との被りチェック（メッセージレベル）を初稿提案前に自己実施**する
- **`rules/self_check.md` を初稿提出前に通す**
- 初稿は「依頼者が微調整指示するだけで完成形に近い」レベルを目標とする

## ファイル運用ルール

- `draft/{{投稿日}}/content.txt` には**検討過程（全パターン）をそのまま残す**。クリーンアップしない。
- 確定版は `posted/normal_post/{{投稿日}}.txt` または `posted/calendar_post/{{投稿日}}.txt` に生成する。
- **日付フォーマットはハイフンつなぎ `YYYY-MM-DD` で統一する**（例：`2026-03-07.txt`）。
- **投稿文面のファイル形式は `.txt`（プレーンテキスト）とする**。`.md` では `(*^^*)` などの記号がマークダウン記法として解釈される。
